# White Label Mini Apps — решения по аудиту от 18.07.2026

Источник: внешний аудит `CHIZHEVSKAYA_PRO_ACTUAL_TZ_AND_AUDIT_2026-07-18.md`. Этот файл фиксирует, какие рекомендации перенесены в проект, а какие осознанно отложены.

## Внесено в текущий релиз

- Сохранён существующий URL `/services/mini-apps-for-bloggers` и общая архитектура сайта.
- Каталог работает по схеме «профессия → релевантные идеи → отдельная карточка решения».
- Сохранены 44 идеи другого агента и добавлены сценарии клиники: анкета до консультации, план после консультации, кабинет с ролями.
- Добавлены отдельный экземпляр приложения и отдельная БД для первых медицинских клиентов.
- Концепты помечены как «концепция интерфейса», а не как готовые работающие продукты.
- Подготовлены семь разных комплектов CSS-мокапов по 3 экрана.
- AI остаётся функцией внутри выбранного приложения.
- Второе/третье приложение — отдельный проект; Web SaaS — дополнительный дорогой формат.
- CTA каталога передаёт профессию и название приложения в единый экран Telegram / MAX / VK.
- Добавлены события аналитики выбора профессии, решения и канала связи.
- Исправлены `Service`, `FAQPage` и `BreadcrumbList` schema, title, description и Open Graph image.
- Видимые цены согласованы с `AggregateOffer`: учитываются только два тарифа с числовой ценой.

## Отложено осознанно

- Перенос страницы на `/services/white-label-mini-apps`: пользователь попросил пока не менять текущую архитектуру и URL.
- 7–8 отдельных нишевых SEO-страниц: сначала выпускается одна сильная каталоговая страница.
- Реальные кликабельные демо: текущие визуалы честно подписаны как примеры интерфейса.
- Массовый контент-кластер блога и отдельные страницы продуктов/кейсов: следующий этап после приёмки каталога.
- Удаление других AI-услуг со всего сайта: требует отдельного решения по общему позиционированию, поэтому не включено в локальную правку страницы White Label.

## Следующий безопасный этап

1. Собрать данные по выбору профессий и решений в аналитике.
2. Сделать реальные прототипы для 2–3 самых востребованных ниш.
3. Создавать нишевые посадочные только при наличии уникальных мокапов, кейсов и поискового спроса.
4. Перед медицинским запуском отдельно проверить обработку персональных данных, согласия, роли, журнал доступа, сроки хранения и резервное копирование под нужную юрисдикцию.
